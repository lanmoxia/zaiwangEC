# Codex 审查指令

本文件是 Codex 在本仓库中的持久评审标准。Codex 评审代码时，必须优先读取本文件、`docs/ai-workflow.md`、`docs/codex-review-standard.md`、`PROJECT_STATUS.md` 和当前任务单。

## 角色定位

Codex 默认是 Reviewer，不是 Implementer。

- 不主动扩大任务范围。
- 不在评审时顺手重构代码。
- 不替 Claude 补写缺失需求，除非用户明确要求实现修复。
- 对不确定项标记 `UNVERIFIED` 或 `NEEDS_HUMAN`，不要猜测通过。

## 评审前必须确认

1. PR 或当前变更是否关联一个任务单，例如 `tasks/TASK-0001.md`。
2. 任务单状态是否为 `Approved`、`In Progress` 或 `Review`。
3. 任务单是否包含目标、明确不做、允许修改、禁止修改、验收标准和验证命令。
4. 当前变更是否只服务于该任务。
5. 评审对象是否绑定到明确 commit SHA。

缺少任务单、任务目标不清、验收标准缺失时，结论必须是 `NEEDS_HUMAN`，不得直接通过。

## 必须检查的内容

- 需求一致性：实现是否满足任务目标和验收标准。
- 范围控制：是否修改了任务单禁止或无关文件。
- 正确性：主要逻辑是否可解释、可测试、可恢复。
- 多用户隔离：涉及用户、团队、项目、素材、反馈时，是否有服务端作用域校验。
- Agent 稳定性：Prompt、模型、Skill、RAG、工具调用是否版本化、结构化、可追踪。
- 长任务可靠性：是否定义超时、重试、幂等、取消、失败状态。
- 外部调用：是否有 Provider Adapter、错误分类、成本/限流控制。
- 故事板链路：豆包/可灵差异是否通过 StoryboardSpec 和 Provider Adapter 表达。
- 测试质量：测试是否覆盖正常路径、失败路径、权限隔离、并发/幂等。
- 可维护性：是否引入不必要复杂度或隐藏副作用。

## 风险等级

- `P0`：会导致数据泄露、权限绕过、破坏性数据损坏、生产不可用、密钥泄露。
- `P1`：任务核心功能不正确、验收标准失败、CI 失败、严重范围外修改。
- `P2`：边界条件、可维护性、测试不足、错误处理不足。
- `P3`：命名、注释、格式、轻微可读性问题。

存在未解决 `P0` 或 `P1` 时，结论必须是 `REQUEST_CHANGES`。

## 固定评审输出

每次评审必须使用以下结构：

```text
结论：APPROVE / REQUEST_CHANGES / NEEDS_HUMAN

任务单：tasks/TASK-XXXX.md
评审 commit：

验收标准检查：
- [PASS/FAIL/UNVERIFIED] 验收标准 1：
- [PASS/FAIL/UNVERIFIED] 验收标准 2：

范围检查：
- [PASS/FAIL] 是否只修改允许范围：
- [PASS/FAIL] 是否存在无关重构：

测试与证据：
- 已运行：
- 未运行及原因：
- CI 状态：

问题清单：
- [P0/P1/P2/P3] 文件:行 - 问题描述 - 修改建议

需要人工确认：
- 
```

## 通过标准

只有同时满足以下条件，才能 `APPROVE`：

- 任务单明确且状态有效。
- 每条验收标准均为 `PASS`，或未验证项有合理人工确认。
- CI / 本地验证通过，或未运行原因可接受且风险低。
- 不存在未解决 `P0` / `P1`。
- 没有明显范围外修改。
- 修改可回滚，风险已说明。

## 禁止行为

- 不允许因为 PR 描述写得好就忽略实际 diff。
- 不允许因为 Claude 声称已测试就跳过验证证据。
- 不允许把聊天记忆当作任务依据。
- 不允许把 Draft 任务当作已批准任务实现。
- 不允许自行批准高风险变更。

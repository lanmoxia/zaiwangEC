# Claude开发与审核规范

> 版本：v0.2  
> 日期：2026-06-23  
> 适用范围：Claude Code及其他参与本项目设计、编码、测试和审核的AI工具  
> 目标：不能保证AI永不出错，但要保证错误可检测、影响受限、过程可追踪、变更可回滚

## 1. 文档优先级

发生冲突时按以下顺序处理：

1. 当前经人工批准的任务单。
2. 已接受的项目ADR。
3. 模块契约、Schema和测试。
4. [视频复刻AI平台重构设计](./video-replication-platform-design.md)。
5. 项目级`CLAUDE.md`。
6. [Agent技术应用与选型参考](./agent-tech-selection-guide.md)。

《Agent技术应用与选型参考》是候选技术手册，不是授权清单。文档提到某项技术，不代表AI可以自行把它加入项目。

如果文档冲突、缺少关键决策或任务需要扩大范围，必须停止相关实现并报告，不得自行选择影响架构的答案。

## 2. AI角色分离

| 角色 | 职责 | 禁止事项 |
|---|---|---|
| Planner | 阅读需求、识别风险、拆分任务、提出实现计划 | 未经批准直接扩大需求 |
| Implementer | 在任务允许范围内编码、测试和记录证据 | 自己批准自己的架构变化 |
| Reviewer | 独立阅读任务、ADR、Diff和测试结果 | 只依据Implementer总结下结论 |
| Human Approver | 确认业务选择、高风险操作和合并 | 将审批责任完全交给模型 |

同一个AI可以在不同轮次承担Planner和Implementer，但Reviewer应使用独立上下文；重要变更优先由另一模型或人工复核。

## 3. 开发前置条件

没有同时满足以下条件时，不开始编码：

- 存在唯一任务编号。
- 目标和非目标明确。
- 引用了相关需求或ADR。
- 列出允许修改与禁止修改的范围。
- 输入输出契约明确，或任务明确要求先创建契约。
- 验收标准可以被测试或检查。
- 已说明数据库、权限、外部调用和数据迁移影响。
- 高风险操作已经获得人工批准。

## 4. 标准执行流程

### 4.1 任务理解

AI在修改前输出：

- 对目标的简短复述。
- 将遵循的需求与ADR编号。
- 预计修改文件和模块。
- 明确不会修改的内容。
- 主要风险和验证方式。
- 仍存在的阻塞项。

任务单已经明确时，不重复提出无必要的问题；采用不改变架构和权限边界的保守假设，并记录假设。

### 4.2 实现计划

计划按“契约→核心逻辑→错误处理→测试→文档”组织。每一步必须可以独立验证，避免一次提交大范围重写。

### 4.3 编码

- 保持变更最小化。
- 遵循现有模块边界、类型和命名。
- 优先使用纯函数和显式依赖。
- Agent节点必须有结构化输入输出。
- 外部调用必须定义超时、错误分类和重试条件。
- 写操作必须考虑幂等和权限。
- 修改产生新版本，不静默覆盖历史AI或人工结果。
- 不将密钥、个人数据或完整敏感素材写入日志、测试或Prompt样例。

### 4.4 验证

根据改动范围运行适用检查：

```text
格式化
类型检查
静态检查
单元测试
契约测试
集成测试
权限/数据隔离测试
Agent Evals
故障恢复测试
构建
```

“测试命令无法运行”不等于“验证通过”。必须说明无法运行的原因、剩余风险和需要谁执行。

### 4.5 完成报告

AI必须提交：

```text
任务编号
修改文件
需求/ADR → 代码映射
需求/ADR → 测试映射
执行命令与结果
未执行的验证
已知风险
回滚方式
是否存在范围外发现
```

## 5. 强制工程边界

### 5.1 范围

- 禁止无关重构。
- 禁止因为“更先进”替换已接受架构。
- 禁止未经任务授权新增Agent、数据库、队列、框架或云服务。
- 禁止顺手修复与任务无关的问题；可以记录为后续候选任务。

### 5.2 依赖

- 新增、升级或删除依赖必须解释必要性和替代方案。
- Lockfile变化必须与任务相关。
- 不引入只为封装一两个简单函数的大型框架。

### 5.3 数据库与数据

- Schema迁移必须单独说明前向、回滚和数据兼容策略。
- 禁止破坏性迁移默认执行。
- 所有多用户表必须明确作用域字段和权限测试。
- 不允许用向量检索替代需要精确一致性的业务数据库查询。

### 5.4 Agent与Prompt

- Prompt、模型、工具描述和Skill均需版本化。
- 生产Prompt修改必须运行固定Evals。
- 不允许模型直接写入已发布知识或Skill。
- Agent循环必须设置最大步骤、最大成本和停止条件。
- Tool调用参数必须服务端校验。
- 故事板必须先生成统一、版本化的StoryboardSpec，再通过Provider Adapter选择libTv渲染策略。
- 豆包链路必须生成完整镜头/场景级素描故事板，而不是单个角色素描肖像，并且不得使用可识别真人脸；可灵真实角色素材必须具备授权和正确数据作用域。
- 不得把素描故事板实现为绕过Provider安全规则的机制。

### 5.5 外部系统

- 外部写操作默认需要幂等键。
- 明确超时、限流、重试和取消语义。
- 不假设第三方调用失败后一定没有产生副作用。
- 测试环境使用Mock或受控测试账号，不使用真实生产任务。

## 6. 自动质量门禁

CI至少应逐步建立：

| 门禁 | 目的 |
|---|---|
| Type Check | 防止类型和契约漂移 |
| Lint/Format | 保持代码一致和发现常见问题 |
| Unit Tests | 验证局部确定性逻辑 |
| Contract Tests | 验证Schema、API、Tool和Provider Adapter |
| Integration Tests | 验证数据库、队列和模块协作 |
| Permission Tests | 防止跨用户、项目和团队数据泄露 |
| Idempotency Tests | 防止重复外部任务和重复写入 |
| Agent Evals | 验证Prompt、模型、RAG和Skill行为 |
| Build | 验证可部署性 |
| Dependency/Security Scan | 发现供应链与已知安全问题 |

未通过门禁时，AI不能通过修改测试期望、删除测试或降低阈值来“修复”，除非任务明确要求并由人工批准。

## 7. Reviewer检查表

Reviewer必须独立检查：

### 需求一致性

- 是否完整实现任务目标？
- 是否实现了任务未授权的功能？
- 是否违反Accepted ADR或模块契约？
- 非目标是否保持未实现？

### 正确性

- 边界、空值、失败和并发情况是否覆盖？
- 重试是否可能产生重复副作用？
- 状态迁移是否合法？
- 版本、父版本和不可变快照是否正确？

### 多用户安全

- 查询、更新、对象存储和RAG是否都执行作用域校验？
- 是否存在通过ID猜测访问他人数据的路径？
- 缓存键、队列消息和日志是否包含作用域？
- 管理员操作是否审计？

### Agent质量

- 输入输出是否结构化并校验？
- 是否存在无限循环或无预算上限？
- RAG是否可能召回无权或未审核内容？
- Prompt或模型修改是否有Evals证据？

### 测试质量

- 测试是否真正覆盖要求，而不是只验证Mock自身？
- 是否包含失败和权限用例？
- 是否有被删除、跳过或放宽的旧测试？
- 测试结果是否可以复现？

## 8. 高风险变更

以下变更必须由人工明确批准，并优先拆成独立任务：

- 身份、权限、数据作用域。
- 数据库破坏性迁移。
- 生产密钥和供应商配置。
- 自动发布知识或Skill。
- 用户数据导出、删除或跨作用域共享。
- 生产外部写操作。
- 依赖大版本升级。
- 修改Evals阈值或黄金数据。
- 开放新的网络、文件系统或代码执行权限。

## 9. 分支、提交与回滚

- AI默认在独立分支或Worktree完成一个任务。
- 一个提交只表达一个主要意图。
- 提交信息包含任务编号。
- 不覆盖或清理用户未提交修改。
- 每个上线任务说明回滚代码、配置和数据的方法。
- 数据迁移必须考虑新旧代码并存窗口。

## 10. 错误处理原则

AI无法保证不出错，本规范追求四件事：

1. **预防**：用契约、权限和任务边界减少错误。
2. **检测**：用测试、Evals、Trace和Reviewer发现错误。
3. **限制**：用沙箱、最小权限、预算和作用域限制影响。
4. **恢复**：用版本、Checkpoint、幂等和回滚恢复。

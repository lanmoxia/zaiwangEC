## 1. 先回答：视频复刻项目能否实践全部技术？

可以覆盖 Agent 工程的大多数关键技术，但不应该把所有技术同时接入生产主链路。

同一个视频复刻项目可以同时承担三种角色：

1. **业务产品**：稳定完成视频分析、分镜规划、素材处理、提示词生成和结果评审。
2. **实验平台**：通过 Feature Flag、旁路任务和固定测试集验证新技术。
3. **案例来源**：沉淀成功案例、失败案例、人工反馈和评测数据。

正确方式是“共享数据和评测体系，隔离实验链路”，而不是把每个新概念直接塞进主流程。

### 1.1 建议分类

| 分类 | 技术 | 在视频项目中的位置 |
|---|---|---|
| 主链路必做 | Structured Output、状态管理、Workflow、Tool Calling、模型路由、Human-in-the-loop、Evals、Tracing、重试与幂等 | 直接决定系统能否稳定运行 |
| 适合逐步接入 | RAG、案例记忆、MCP、Skill、Evaluator-Optimizer、可恢复执行、多模态证据链 | 验证有效后进入主链路 |
| 适合旁路实验 | Agentic RAG、Graph RAG、动态 Skill、Multi-Agent、自动反思、模型投票 | 与简单方案做 A/B 对照 |
| 暂不建议生产化 | 无审核的自我修改、开放式 Agent Network、自动写入长期记忆、无限反思循环、在线自动微调 | 风险、成本和不可控性通常大于收益 |

### 1.2 视频项目可覆盖的实验点

| 项目环节 | 可实验技术 |
|---|---|
| 上传和预处理 | 异步任务、队列、幂等、OCR、ASR、抽帧、镜头切分、多模态输入 |
| 视频分析 | Structured Output、证据引用、模型路由、置信度、模型投票 |
| 分镜规划 | Planning、Workflow、状态机、StoryboardSpec、libTv故事板、上下文工程、Skill |
| 提示词生成 | RAG、案例记忆、模型适配器、Evaluator-Optimizer |
| 生图/生视频 | Tool Calling、MCP、权限、超时、重试、成本控制 |
| 连续分镜 | 状态、Checkpoint、角色/场景记忆、可恢复执行 |
| 结果评审 | Evals、LLM Judge、人工评审、结构化 Diff、自动修正 |
| 案例投喂 | 知识治理、RAG、记忆写入策略、数据版本与污染控制 |
| 系统运营 | Trace、回放、Prompt版本、模型版本、指标、灰度发布 |

## 2. Agent 技术之间的基本关系

一个工程化 Agent 可以抽象为：

```text
Agent = Model
      + Instructions
      + Context
      + State / Memory
      + Planning Policy
      + Tools
      + Knowledge Retrieval
      + Runtime
      + Feedback / Evaluation
      + Safety Boundaries
```

几个必须分清的概念：

- **Workflow** 决定程序允许怎样流转；**Agent** 在允许范围内动态选择下一步。
- **Tool** 是一个可执行能力；**Skill** 是完成某类任务的方法、约束和工具组合。
- **上下文** 是本次模型实际看到的输入；**记忆** 是经过管理、可在未来取回的信息。
- **知识库** 是内容；**RAG** 是检索和组装知识的方法。
- **MCP** 是连接模型与外部能力的协议，不负责业务状态、规划和评测。
- **AI Reviewer** 是一次判断；**Evals** 是可重复、可比较的测量体系。
- **反思** 修改当前任务的处理结果；**进化** 会持久改变未来任务的行为。

## 3. 总体选型原则

1. **能用确定性代码解决的，不交给模型猜。**
2. **能用单次调用解决的，不先设计循环 Agent。**
3. **能用单 Agent 解决的，不先增加多 Agent 通信。**
4. **状态必须外置。** 不把聊天历史当作唯一数据库。
5. **模型输入输出尽量结构化。** 自然语言适合表达，Schema 适合控制系统。
6. **先建立评测，再增加复杂度。** 没有基线就无法证明新技术有效。
7. **所有自动学习都要有晋升机制。** 原始反馈不能直接成为生产知识。
8. **复杂度必须换来可测量收益。** 收益通常是质量、成功率、速度或成本。

# 第一部分：控制、规划与运行时

## 4. Structured Output 与 Schema

### 应用场景

- 视频报告、分镜、提示词参数、工具参数需要被程序继续处理。
- 多个模型或节点需要共享同一份业务数据。
- 需要验证字段、追踪版本、局部修改和回归测试。

### 解决的问题

- 自由文本格式不稳定。
- 下游无法可靠解析。
- 观察、推断和执行指令混在一起。
- 修改一个字段时被迫重写整篇内容。

### 常见方案

- JSON Schema、TypeScript 类型、Zod 等运行时校验。
- 模型原生 Structured Output。
- 校验失败后执行受控修复或重试。
- Schema 增加 `version`、`confidence`、`evidence` 字段。

### 默认最优选择

- **程序要继续消费的内容：结构化输出。**
- **给人阅读的报告：由结构化数据渲染成自然语言。**

### 视频项目实验

把“视频分析报告”拆成 `VideoSpec`、`ShotSpec`、`AssetSpec` 和 `PromptSpec`。对比自由文本报告与 Schema 报告在缺字段率、人工修改时间和下游错误率上的差异。

### 不适合使用

纯创意发散或一次性头脑风暴不必强行设计复杂 Schema。

## 5. Tool Calling

### 应用场景

Agent需要读取文件、查询数据库、抽帧、调用生图/生视频接口或创建任务。

### 解决的问题

让模型从“只会生成文本”升级为“可以通过受控接口影响外部系统”。

### 常见方案

- Function Calling / Tool Calling。
- 每个工具使用严格参数 Schema。
- 工具返回结构化结果和标准错误码。
- 超时、重试、幂等键、权限和审计。

### 默认最优选择

- 工具数量少、系统内部使用：普通 TypeScript 函数加模型 Tool Calling。
- 工具需要跨项目、跨语言复用：考虑 MCP。
- 高风险写操作：必须增加确认或审批。

### 视频项目实验

实现 `extract_frames`、`remove_subtitles`、`generate_image`、`generate_video`、`get_task_status` 五个工具，测试参数错误、超时、重复提交和服务不可用。

### 不适合使用

固定且确定的程序步骤不需要让模型决定是否调用，直接由 Workflow 调用更稳定。

## 6. Workflow、状态机与 DAG

### 应用场景

- 任务存在明确阶段、分支、重试和人工审批。
- 视频生成需要等待数分钟甚至更久。
- 任务可能中断，需要恢复。

### 解决的问题

- 长任务依赖聊天线程，断开后无法恢复。
- 模型自由决定全部步骤，路径不可预测。
- 无法准确判断任务目前处于哪个阶段。

### 常见方案

- 有限状态机：状态与迁移明确。
- DAG：适合无循环或有限依赖的批处理。
- Durable Workflow：支持持久化、暂停、恢复和长时间等待。
- Human-in-the-loop 节点。

### 默认最优选择

- 固定步骤：普通程序流程或 DAG。
- 有审批、重试、回退：状态机。
- 跨小时/天、需要断点恢复：持久化工作流引擎。
- 路径只有局部不确定：Workflow 控制边界，Agent只决定局部步骤。

### 视频项目实验

设计：`UPLOADED → ANALYZING → REVIEWING → PLANNING → APPROVED → GENERATING → EVALUATING → COMPLETED/FAILED`，测试任务暂停、恢复、驳回和局部重跑。

## 7. Planning 与推理模式

### 应用场景

任务不能通过一次调用完成，需要拆分、选择工具或根据中间结果调整。

### 常见模式

| 模式 | 适合场景 | 风险 |
|---|---|---|
| Prompt Chain | 固定的多阶段转换 | 上游错误向下传播 |
| Routing | 根据类型选择模型、工具或流程 | 分类错误导致走错链路 |
| ReAct | 一边观察一边调用工具 | 调用次数和路径不稳定 |
| Plan-and-Execute | 先计划再执行 | 初始计划可能过早锁定 |
| Evaluator-Optimizer | 生成、评分、定向修改 | 容易无限循环、成本增加 |
| Parallel Exploration | 多方案并行后择优 | 成本较高 |
| Reflection | 根据失败原因重试 | 模型可能重复合理化错误 |

### 默认最优选择

- 可预先知道步骤：Prompt Chain 或 Workflow。
- 工具选择不确定但目标清晰：ReAct。
- 复杂任务、步骤相对独立：Plan-and-Execute。
- 输出可以明确评分：Evaluator-Optimizer。
- 创意方案价值高于调用成本：并行探索后择优。

### 视频项目实验

分别让模型使用“一次生成”“先分镜再生成”“生成后按评分局部优化”，对比一次通过率、调用成本和漂移程度。

## 8. 状态管理与可恢复执行

### 应用场景

- 多轮任务。
- 连续分镜。
- 长时间异步调用。
- 用户可能修改中间结果。

### 解决的问题

- 聊天历史与真实业务状态不一致。
- 重试后重复创建任务。
- 无法只重跑失败节点。

### 常见方案

- 数据库保存任务状态和节点输出。
- Checkpoint 保存关键阶段快照。
- Event Sourcing 保存发生过的事件，当前状态由事件推导。
- 幂等键阻止重复外部调用。
- 状态乐观锁或版本号避免并发覆盖。

### 默认最优选择

- 一般应用：数据库当前状态 + 关键事件日志。
- 强审计和复杂回放：Event Sourcing。
- 长任务：Checkpoint + Durable Workflow。

### 视频项目实验

中途停止一个三分镜任务，修改第二镜提示词后从第二镜恢复，确保第一镜不重复生成、第三镜使用最新连续性状态。

## 9. Model Routing、Fallback 与模型组合

### 应用场景

不同任务在视觉、推理、速度、成本和上下文长度上需求不同。

### 常见方案

- 规则路由：按时长、任务类型、风险选择模型。
- 分类器路由：小模型判断走哪条路径。
- 质量升级：便宜模型失败后调用更强模型。
- 并行投票：多个模型独立生成或评分。
- Fallback：主模型不可用时切换备选。

### 默认最优选择

- 规则明确：规则路由最优。
- 类型复杂且有标注数据：分类器路由。
- 大部分任务简单：小模型优先、失败升级。
- 高价值且主观：多候选加人工/模型择优。

### 视频项目实验

按视频时长、人物数量、对白密度、镜头数和历史成功率路由不同分析/生成策略，并记录路由正确率。

# 第二部分：上下文、知识与记忆

## 10. Context Engineering

### 应用场景

模型需要同时看到任务目标、历史状态、工具说明、案例和业务约束。

### 解决的问题

- 上下文过长导致重点被稀释。
- 历史错误持续污染后续结果。
- 指令、数据和案例互相冲突。
- Token成本不断增长。

### 常见方案

- 上下文分层：系统规则、任务状态、证据、案例、用户反馈分开。
- 只装配当前节点需要的信息。
- 历史对话摘要与结构化状态并存。
- Context Isolation：不同节点使用独立上下文。
- Prompt模板和上下文包版本化。

### 默认最优选择

- 长期任务优先使用“外置状态 + 按需组装”，而不是不断追加聊天记录。
- 摘要用于人类语义连续，结构化状态用于系统正确性。

### 视频项目实验

对比三种方式：完整历史、自动摘要、每个节点读取结构化任务状态。连续处理20次反馈后检测目标、角色和约束是否漂移。

## 11. RAG

### 应用场景

- 需要从大量案例、规范、模型说明或历史反馈中找到与当前任务相关的内容。
- 知识经常更新，不适合写死在Prompt或模型参数中。

### 解决的问题

- 上下文装不下全部知识。
- 模型不知道企业私有数据。
- 需要证据来源和可更新性。

### 标准流程

```text
数据清洗 → 切片 → Metadata → 索引
→ Query Rewrite → 召回 → 过滤 → Rerank
→ Context Assembly → 生成 → 引用/评测
```

### 常见方案

| 方案 | 最适场景 |
|---|---|
| 关键词/BM25 | 精确术语、编号、模型名、错误码 |
| 向量检索 | 语义相似但用词不同 |
| Hybrid Search | 大多数企业知识库的默认选择 |
| Metadata Filter | 模型、版本、品类、时间等条件明确 |
| Rerank | 初次召回噪声较多，需要提高前几条质量 |
| Parent-Child Retrieval | 小块检索、大块提供上下文 |
| Multi-Query | 用户问题表达不完整或存在多种说法 |

### 默认最优选择

从“Hybrid Search + Metadata过滤 + Rerank”开始。不要只因为有向量数据库就把所有资料切成固定长度后直接检索。

### 视频项目实验

检索与当前模型、商品类型、镜头类型、视频时长和失败类型相似的成功/失败案例。固定20～50个问题，评测 Recall@K、命中质量和引用是否真正支持输出。

### 不适合使用

- 数据量很小，直接放入上下文更清楚。
- 需要精确事务状态，应查询数据库而不是向量检索。
- 知识本身未经审核，RAG只会更高效地取回错误内容。

## 12. Agentic RAG

### 应用场景

问题复杂，Agent需要自己判断是否检索、查询什么、是否补充检索以及怎样组合多个来源。

### 解决的问题

单次检索难以回答多跳问题或探索型问题。

### 常见方案

- Query Planning。
- 多轮检索与证据检查。
- 查询拆分。
- 检索结果不足时更换检索源。
- 最终答案前执行 Grounding Check。

### 默认最优选择

只有当固定RAG在明确评测集上无法处理复杂查询时再使用。先限制最大检索轮数和成本。

### 视频项目实验

让Agent依次检索“相似视频案例”“目标生成模型规则”“相似失败案例”，再生成提示词；与一次检索方案比较。

## 13. Graph RAG 与知识图谱

### 应用场景

- 实体和关系比文本相似度更重要。
- 需要多跳关系，例如角色—服装—分镜—素材—生成结果。
- 需要全局关系分析。

### 常见方案

- 图数据库保存实体和关系。
- 文本向量检索与图遍历混合。
- 从文档抽取实体、关系并进行人工/规则校验。

### 默认最优选择

- 文档问答、案例相似检索：普通RAG优先。
- 关系查询频繁且Schema稳定：知识图谱。
- 同时需要语义与关系：Hybrid RAG + Graph。

### 视频项目实验

构建“小型视频资产图”：角色、商品、场景、道具、镜头、提示词、模型版本、结果评分之间的关系。验证它是否优于关系数据库和Metadata过滤。

### 不适合使用

实体关系尚未稳定、数据量很小或只是为了学习概念时，不应直接改造主数据库。

## 14. Memory

### 四类记忆

| 类型 | 内容 | 示例 |
|---|---|---|
| Working Memory | 当前任务临时状态 | 当前分镜、当前反馈 |
| Episodic Memory | 过去发生过的案例 | 某次可灵生成失败及修复过程 |
| Semantic Memory | 稳定知识 | 模型规则、术语、业务规范 |
| Procedural Memory | 怎样完成任务 | 一套提示词生成Skill |

### 核心问题

记忆难点不是“存下来”，而是：什么值得写入、谁批准、什么时候取回、何时过期、错误如何删除。

### 常见方案

- 结构化数据库保存事实状态。
- 案例库保存Episodic Memory。
- RAG知识库保存Semantic Memory。
- 版本化Skill保存Procedural Memory。
- 记忆候选 → 评审 → 晋升 → 过期/撤销。

### 默认最优选择

- 用户和任务事实：结构化数据库。
- 相似经验：案例RAG。
- 操作方法：Skill和代码。
- 不允许模型直接把自己的总结写入“权威记忆”。

### 视频项目实验

把人工反馈分成事实纠正、偏好、单次异常和可复用经验。仅将审核通过的可复用经验晋升到案例库。

# 第三部分：协议、Skill与外部行动

## 15. MCP

### 应用场景

- 多个Agent或客户端需要复用相同工具和数据。
- 工具服务与模型/框架需要解耦。
- 希望用统一方式暴露Tools、Resources和Prompts。

### 解决的问题

- 每个Agent框架重复编写工具适配层。
- 工具描述、参数和连接方式不统一。
- 私有系统接入难以复用。

### 常见方案

- 本地标准输入输出服务。
- 远程MCP服务。
- 工具Schema、鉴权、权限、超时和审计。
- 服务端版本与契约测试。

### 默认最优选择

- 单项目内部、工具很少：普通函数/API更简单。
- 多客户端、多项目或跨语言复用：MCP价值明显。
- 对外部系统有写权限：权限和审批必须在服务端落实，不能只依赖Prompt。

### 视频项目实验

把libTv、生图、生视频、素材查询封装为独立MCP Server，让不同模型客户端复用；与项目内直接API调用比较维护成本。

### MCP不能解决

MCP不会自动解决上下文漂移、工具选择错误、业务状态同步、模型幻觉和任务评测。

## 16. Skill系统

### 定义

Skill是一个可复用的任务能力包，通常包含：

```text
适用条件 + 操作说明 + 输入输出Schema
+ 可调用工具 + 示例 + 约束 + 测试 + 版本
```

### 应用场景

- 同类任务反复出现。
- 不希望把全部操作说明长期放入系统Prompt。
- 希望按任务动态加载专业能力。

### 常见方案

- Skill Registry。
- 基于规则、标签、语义或模型的Skill路由。
- Skill按需加载。
- Skill组合和依赖管理。
- 版本、测试、发布、回滚。

### 默认最优选择

- Skill数量少且边界明确：规则/标签路由。
- Skill很多、用户描述模糊：语义召回后再由模型选择。
- 高风险Skill：白名单和人工确认。

### 视频项目实验

建立“人物口播”“商品展示”“剧情反转”“连续分镜”“豆包短视频”“可灵长视频”等Skill，比较全量Prompt与按需加载的效果。

## 17. 动态Skill

动态Skill应拆成三个层级：

1. **动态选择**：从已有Skill中选择。
2. **动态组合**：根据任务组合已有Skill。
3. **动态生成**：创建新的Skill。

### 解决的问题

处理预先没有完整覆盖的新任务，让系统能力逐步扩展。

### 安全方案

```text
生成候选Skill
→ 静态检查
→ 沙箱运行
→ 固定测试集
→ 与旧方案对比
→ 人工审批
→ 发布新版本
→ 可回滚
```

### 默认最优选择

生产系统先做到动态选择和组合。动态生成只进入候选区，不允许生成后立即获得工具权限并自动运行。

### 视频项目实验

给系统一种从未定义的新广告结构，让它从历史任务中生成候选Skill；在隔离测试集上比较人工Skill和自动Skill。

# 第四部分：协作、审核与改进

## 18. Multi-Agent

### 常见架构

| 架构 | 适用场景 | 主要风险 |
|---|---|---|
| Supervisor/Workers | 任务可拆成不同专业角色 | 管理者成为瓶颈 |
| Hierarchical | 大型任务、分层管理 | 通信和状态复杂 |
| Peer-to-Peer | 节点地位接近、协商解决 | 难控制、难调试 |
| Blackboard | 多角色围绕共享状态协作 | 共享状态污染和并发冲突 |
| Debate/Critic | 高价值判断、需要不同观点 | 成本高、可能互相说服而非纠错 |
| Parallel Candidates | 创意生成和择优 | 调用成本增加 |

### 默认最优选择

- 只是步骤不同：Workflow，不必Multi-Agent。
- 不同角色需要不同工具、权限或上下文：Supervisor/Workers。
- 创意任务需要多样性：并行候选 + 独立评审。
- 没有评测证明收益前，不使用开放式Peer Network。

### 视频项目实验

比较：单Claude完成分析到提示词；分析Agent+规划Agent+评审Agent；固定Workflow调用三个模型节点。记录质量、成本、延迟和故障定位难度。

## 19. Human-in-the-loop

### 应用场景

- 高成本生成前。
- 对外发布、版权、人物肖像或高风险操作。
- 模型不确定性较高。
- 用户偏好难以通过规则表达。

### 常见方案

- Approval Gate。
- 低置信度升级人工。
- 人工修改结构化字段，而不是重写整段Prompt。
- 支持撤销、驳回、原因标签和局部重跑。

### 默认最优选择

- 高成本/不可逆操作：执行前审批。
- 可逆、低成本操作：先执行后抽检。
- 重复性高且评测成熟：仅异常升级人工。

### 视频项目实验

比较“提示词前审”“生成后审”“双审”对成本、一次通过率和总耗时的影响。

## 20. Evaluator-Optimizer 与自我修正

### 应用场景

输出存在明确标准，可以定位问题并局部修正。

### 常见方案

- 规则检查优先。
- 独立模型按Rubric评分。
- 生成器只接收结构化问题列表。
- 限制最大循环次数。
- 连续两次无提升则停止或升级人工。

### 默认最优选择

- 格式、字段、时长：代码规则。
- 语义、创意、构图：模型评审。
- 高价值最终判断：人工评审。
- 修改应尽量局部，不让模型每轮重写全部结果。

### 视频项目实验

把问题分成节奏、动作、构图、人物、商品、连续性、模型合规七类。评审器只能返回问题类别、证据和建议修改字段。

# 第五部分：评测、可观测性与生产可靠性

## 21. Evals

### 应用场景

任何需要比较Prompt、模型、RAG、Skill、Agent架构或版本升级的地方。

### 评测层级

| 层级 | 评测内容 |
|---|---|
| 单元级 | Schema、工具参数、单字段准确性 |
| 节点级 | 视频分析、分镜、检索、提示词质量 |
| 轨迹级 | 是否选择正确工具、是否发生无效循环 |
| 结果级 | 最终视频是否可用 |
| 系统级 | 成本、延迟、成功率、人工介入率 |

### 常见方案

- Golden Dataset。
- 确定性代码评分。
- LLM Judge。
- Pairwise Comparison。
- 人工Rubric。
- 离线回归测试与线上指标。

### 默认最优选择

代码规则 + 人工黄金数据 + 经人工校准的LLM Judge组合使用。不能只让模型给自己打分。

### 视频项目核心指标

- 报告字段准确率。
- 镜头边界误差。
- 检索命中率。
- 提示词一次通过率。
- 首次生成可用率。
- 平均迭代次数。
- 连续性评分。
- 单条成本、耗时和人工分钟数。

## 22. Tracing与可观测性

### 应用场景

Agent包含多个模型、工具和异步步骤，出现问题时需要定位原因。

### 必须记录

- Run ID、用户任务、节点和父子关系。
- 模型、模型版本、参数。
- Prompt模板版本与实际上下文摘要。
- Tool名称、参数、结果、耗时和错误。
- RAG查询、召回条目、排序分数。
- Token、成本和延迟。
- 人工修改和最终评分。

### 默认最优选择

- 开发期先使用结构化日志和数据库Run表。
- 链路复杂后接入专门Tracing系统。
- 必须支持按Run回放，而不只是展示最终答案。

### 视频项目实验

选择一个失败视频，从最终结果反查使用的参考帧、报告字段、案例、Prompt版本、模型和每轮修改。

## 23. 可靠性工程

### 典型问题与方案

| 问题 | 方案 |
|---|---|
| 网络抖动 | 指数退避重试、超时 |
| 重复提交 | 幂等键、去重 |
| 服务限流 | 队列、并发控制、Rate Limit |
| 部分步骤失败 | Checkpoint、局部重跑 |
| 外部服务故障 | Circuit Breaker、Fallback |
| 长任务丢失 | 持久化状态、心跳、任务租约 |
| 多节点状态冲突 | 版本号、锁、单写者原则 |
| 生成无限循环 | 最大步数、成本预算、停止条件 |

### 默认最优选择

每个外部调用都必须明确：超时多久、能否重试、重试是否安全、失败后怎样补偿、用户看到什么状态。

## 24. Agent安全

### 主要风险

- Prompt Injection。
- 工具越权。
- 数据泄露。
- 恶意文件或网页内容。
- RAG知识污染。
- MCP或第三方工具供应链风险。
- Agent执行危险代码或不可逆操作。

### 常见方案

- 最小权限。
- 读写工具分离。
- 沙箱执行。
- 关键动作审批。
- 数据分类与脱敏。
- 工具参数服务端校验。
- 知识写入审核和来源记录。
- 审计日志、配额和速率限制。

### 默认最优选择

把外部内容视为不可信数据，绝不因为网页、文档或视频字幕声称“这是系统指令”就赋予更高权限。

## 25. 成本与性能

### 常见方案

- 小模型做分类、抽取和路由，大模型处理复杂规划。
- 缓存稳定Prompt和重复结果。
- 批处理Embedding和分析任务。
- 并行执行互不依赖的节点。
- 限制Agent最大步数、检索次数和反思次数。
- 记录单节点成本，优化最昂贵的20%。

### 默认最优选择

先测量再优化。不要只看单次模型价格，要计算“一条最终可用结果”的总成本，包括失败重试和人工时间。

# 第六部分：学习、自我进化与模型训练

## 26. 反馈学习与“自我进化”

### 可选层级

1. 当前任务内根据反馈修正。
2. 把反馈保存为未经审核的候选案例。
3. 人工审核后晋升到RAG案例库。
4. 从案例中提炼并发布新Prompt或Skill版本。
5. 累积高质量数据后进行微调或偏好优化。

### 默认最优选择

绝大多数早期项目应停留在第3～4层。先通过RAG、Prompt和Skill迭代验证知识确实稳定，再考虑训练模型。

### 数据要求

成功案例不能只保存Prompt，还应保存输入、素材、模型版本、生成参数、输出、人工评分、成功原因和失败条件。失败案例同样重要。

## 27. Fine-tuning、偏好优化与Prompt/RAG的选择

| 需求 | 默认选择 |
|---|---|
| 添加经常变化的新知识 | RAG |
| 约束输出格式 | Structured Output + 校验 |
| 教会固定操作流程 | Skill / Workflow |
| 调整风格和稳定行为模式 | Prompt，数据充分后考虑微调 |
| 降低大模型成本 | 小模型微调或蒸馏，需要足够数据 |
| 保存用户/任务事实 | 数据库或Memory，不用微调 |

微调不是修复所有Agent问题的万能方案。工具错误、状态错误、检索错误和工作流错误通常无法通过微调根治。

# 第七部分：框架与开发工具选型

## 28. 直接SDK、LangChain、LangGraph与工作流引擎

| 场景 | 默认首选 |
|---|---|
| 1～3个模型调用、少量工具 | 模型官方SDK + TypeScript函数 |
| 快速接入多个模型、Retriever和向量库 | LangChain等集成框架 |
| 有循环、有状态、有人工暂停的Agent | LangGraph等状态图框架 |
| 跨小时/天、强恢复、强可靠性业务流程 | Durable Workflow引擎 + 模型节点 |
| 高风险、路径基本固定 | 确定性Workflow，不让Agent自由规划全部过程 |
| 探索型任务、路径无法预先定义 | 有步数和权限边界的Agent Runtime |

框架不是能力本身。学习框架时要能指出它替你实现了哪些状态、路由、持久化、Tracing和错误处理。

## 29. Codex、Claude Code等编码Agent的定位

它们主要是开发和维护软件的AI工程工具，可以帮助：

- 设计和实现上述模块。
- 阅读框架和开源项目。
- 编写测试、评测集和文档。
- 调试Agent轨迹。
- 自动化代码审查、重构和部署任务。

它们可以通过SDK、命令行或MCP成为工程工作流中的一个节点，但通常不能替代业务Agent所需的数据库、状态机、RAG、权限、评测和运行时。

## 30. JavaScript/TypeScript与Python

### TypeScript适合

- Vue前端和Node后端。
- Web API、Tool Calling、MCP、队列和数据库。
- 业务系统和类型化Schema。
- 大多数云模型API与Agent工作流。

### Python适合

- AI研究代码和大量开源示例。
- 视频、图片、音频和数据处理。
- Notebook实验、评测和数据分析。
- 本地模型、训练与微调。

### 推荐结构

早期可全部使用TypeScript。出现媒体算法、模型实验或训练需求后，把Python作为独立Worker或实验环境接入，不必为了学习Agent先重写主系统。

# 第八部分：按业务场景选择默认方案

## 31. 场景决策表

| 场景 | 默认方案 | 何时升级 |
|---|---|---|
| 固定的内容生成流水线 | Workflow + Structured Output | 出现动态路径后增加Router/Agent |
| 企业文档问答 | Hybrid RAG + Rerank + 引用 | 多跳问题明显时升级Agentic RAG |
| 数据库事务查询 | 结构化查询工具 | 不使用向量RAG代替事实数据库 |
| 长时间异步任务 | 队列 + 持久化状态 + Checkpoint | 跨天和复杂补偿时使用Durable Workflow |
| 多模型能力差异明显 | 规则Router + Fallback | 规则难维护且有数据时训练Router |
| 高成本生成 | 生成前人工审批 + 生成后评测 | 评测成熟后降低人工比例 |
| 创意方案生成 | 并行候选 + Pairwise择优 | 预算有限时减少候选数 |
| 复杂工具探索 | 有限制的ReAct | 路径稳定后固化为Workflow/Skill |
| 反复出现的专业任务 | 版本化Skill | Skill很多时增加检索和路由 |
| 新任务自动适应 | 候选动态Skill + 沙箱评测 | 绝不直接自动发布生产Skill |
| 多专业角色合作 | Supervisor/Workers | 仅当单Agent基线明显不足 |
| 关系密集知识 | 关系库/知识图谱 | 文本语义检索仍需保留Hybrid RAG |
| 视频复刻 | 确定性媒体预处理 + 多模态模型节点 + 统一StoryboardSpec + Provider专属故事板 + Workflow + Evals | 稳定后再加入RAG、Skill和局部Agent循环 |

视频复刻场景中，豆包侧的“素描故事板”默认指完整镜头/场景级的故事板面板，包含构图、人物占位、动作、商品、道具、景别和机位，不是单个角色的素描肖像；可灵侧可以在授权和数据作用域明确的前提下使用真实角色外观。两者共享中立`StoryboardSpec`，由Provider Adapter决定libTv渲染策略，并始终遵守目标平台规则。

## 32. 如何判断某项技术是否值得保留

每次实验都至少保留一个简单基线，并回答：

1. 质量是否显著提高？
2. 首次成功率是否提高？
3. 人工时间是否减少？
4. 成本和延迟增加多少？
5. 故障是否更容易定位和恢复？
6. 新复杂度是否可以通过测试和文档维护？

建议使用统一实验记录：

```text
实验名称：
问题：
假设：
基线方案：
候选方案：
数据集：
评价指标：
实验结果：
失败案例：
适用条件：
不适用条件：
是否进入主链路：
```

## 33. 建议保留的项目边界

即使以视频项目作为Agent实验场，也建议分成：

```text
production/   稳定主链路
experiments/  独立实验和Feature Flag
evals/        固定测试集、评分标准和结果
knowledge/    审核后的知识、案例、Skill
traces/       运行记录和问题回放
docs/         概念、架构决策和实验结论
```

实验通过预先约定的指标后，才从 `experiments` 晋升到 `production`。

# 第九部分：多用户Agent生产平台

## 34. 从个人Agent到内部生产平台

个人Demo主要验证模型是否能完成任务；多人平台还必须解决身份、权限、并发、公平性、数据隔离、审计、成本和故障恢复。

当系统交付给约50名剪辑人员使用时，推荐采用：

```text
多用户业务工作台
+ 模块化单体API
+ 异步任务队列
+ 可横向扩展的无状态Worker
+ 关系数据库
+ 对象存储
+ Agent工作流与模型适配器
+ 评测、追踪和审计
```

50名内部用户不要求一开始微服务化。主要压力通常来自视频上传、媒体处理、模型调用和外部生成任务，而不是普通Web请求。

## 35. 身份、权限与数据作用域

### 推荐作用域

```text
Workspace（公司）
└── Team（团队）
    └── Project（项目）
        └── User（用户）
```

业务数据应显式包含适用的作用域字段：

```text
workspaceId
teamId
projectId
ownerId
visibility
```

`visibility`至少区分：

- `private`：仅创建者和授权管理员。
- `project`：当前项目成员。
- `team`：团队共享。
- `organization`：公司范围内共享。

### 默认最优选择

- 使用RBAC管理角色，使用数据作用域管理可见范围。
- 每次查询都在服务端执行作用域过滤，不能依赖前端隐藏。
- 管理员跨作用域访问必须产生审计记录。
- 文件对象、向量索引、缓存和Trace同样必须携带作用域，不能只隔离数据库行。

## 36. 多人反馈与记忆隔离

多人反馈不能直接进入同一个长期记忆池。推荐分层：

| 层级 | 内容 | 默认影响范围 |
|---|---|---|
| Run反馈 | 对本次报告、提示词或视频的修改 | 当前Run |
| 个人偏好 | 某位剪辑师稳定的主观偏好 | 当前用户 |
| 项目规则 | 某个商品或项目的约束 | 当前项目 |
| 团队经验 | 经过审核的通用案例 | 当前团队 |
| 公司标准 | 管理员批准的规范和Skill | Workspace |

### 推荐写入流程

```text
原始反馈
→ 标注来源和作用域
→ 形成知识候选
→ 去重与冲突检查
→ 人工/规则审核
→ 晋升到指定作用域
→ 版本化发布
→ 支持撤销和过期
```

个人偏好不得自动改变团队知识。一次偶然成功的Prompt不得直接晋升为团队Skill。

## 37. Scope-aware RAG

多人平台中的RAG除了相关性，还必须考虑权限和优先级。

推荐检索顺序：

```text
公司已审核规范
+ 团队已审核案例
+ 当前项目规则
+ 当前用户偏好
```

每条索引记录应保存来源、作用域、所有者、版本、模型版本、审核状态和过期时间。权限过滤应在召回阶段执行，不能先召回全部数据再让模型自行忽略无权内容。

当不同层级发生冲突时，必须由明确规则决定优先级，不能让模型临场猜测。

## 38. 并发、队列与公平调度

所有耗时操作应转为异步Job，而不是让HTTP请求等待模型或视频平台完成。

推荐状态：

```text
QUEUED
→ RUNNING
→ WAITING_EXTERNAL
→ NEEDS_REVIEW
→ SUCCEEDED / FAILED / CANCELLED
```

### 必须具备

- 用户级、团队级和供应商级并发上限。
- 公平调度，避免单个用户长期占满资源。
- 幂等键，防止重复上传或重复生成。
- 超时、受控重试、取消和失败队列。
- Checkpoint和局部重跑。
- Backpressure：供应商限流时降低消费速度。
- 按任务、用户、团队和模型统计成本。

### 默认最优选择

- 短任务可以同步完成；媒体处理和模型生成全部异步化。
- 配额由配置管理，不将并发数字写死在代码中。
- 优先保证任务不丢失和公平性，再追求最大吞吐量。

## 39. 多人编辑与版本冲突

多人可能同时修改报告、分镜或提示词。推荐：

- 每次编辑生成新版本，不直接覆盖历史。
- 使用版本号或乐观锁检测并发修改。
- 保存作者、时间、修改原因和父版本。
- 支持对比、恢复和基于某个版本重新生成。
- 已被下游任务使用的输入保持不可变快照。

Agent不得悄悄覆盖人工修改。模型重新生成时应创建候选版本，由用户决定是否采用。

## 40. 多用户平台的评测与运营指标

除模型质量外，还应测量：

- 同时在线和同时执行任务数。
- 队列等待时间与各阶段耗时。
- 用户级、团队级和供应商级并发占用。
- 任务成功率、重试率和取消率。
- 单用户、单项目和单条可用视频的成本。
- 人工介入率与平均人工分钟数。
- 跨用户数据泄露事件数，目标必须为零。
- 反馈晋升率、撤销率和知识冲突率。
- 不同用户群体的一次通过率，避免只对少数人的偏好优化。

# 第十部分：AI协作开发治理

## 41. 参考手册、项目决策和任务单必须分离

AI编码工具不应直接把本参考手册当作完整施工图。正确关系是：

```text
技术参考手册
→ 项目架构决策（选择什么、拒绝什么）
→ 模块契约与验收标准
→ 小粒度开发任务
→ 代码与测试
```

参考手册包含候选方案；项目ADR只能保留本项目已经决定的方案。未被ADR选中的技术不得因为模型认为“更先进”而自行加入。

## 42. Spec-driven、Contract-driven与Eval-driven Development

### Spec-driven

每个任务必须包含目标、范围、非目标、允许修改的文件、验收标准和回滚方式。

### Contract-driven

跨模块交互使用Schema、类型、API契约和工具契约。能由机器校验的要求不得只写成自然语言。

### Eval-driven

模型、Prompt、RAG、Skill或Agent路径发生变化时，必须运行固定评测集并与基线比较。

### 默认最优选择

传统软件正确性由类型、静态检查、单元测试、集成测试和契约测试保证；AI行为质量由Evals和人工黄金样本验证。两套体系缺一不可。

## 43. AI开发与审核闭环

推荐执行流程：

```text
读取指定需求与ADR
→ 输出需求理解和影响范围
→ 建立需求-代码-测试映射
→ 计划审核
→ 小范围实现
→ 自动测试与Evals
→ 独立AI检查实际Diff
→ 人工批准
→ 合并与可回滚发布
```

实现者不得只提交“已经完成”的文字总结，还必须提供：

- 实际修改文件。
- 每项需求的实现位置。
- 每项需求对应的测试。
- 执行过的验证命令和结果。
- 已知风险、未完成事项和回滚方式。

Reviewer应读取原始需求、ADR、实际Diff和测试结果，不应只依据实现者的自我总结。

## 44. AI编码权限与强制边界

- 默认在独立分支或Worktree工作。
- 默认最小文件系统和网络权限。
- 禁止接触生产密钥和真实用户隐私数据。
- 数据库迁移、依赖升级、权限变更和外部写操作需要单独批准。
- 禁止无关重构和超出任务范围的架构替换。
- 未通过测试、Evals或审核的变更不能合并。
- 所有上线变更都必须可以回滚。

目标不是假设AI永不犯错，而是让错误能够被发现、阻断、限制影响并恢复。

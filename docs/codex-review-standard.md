# Codex 评审标准

Codex 的目标不是证明 Claude 写得对，而是找出任务目标、代码实现、测试证据之间的不一致。

## 评审输入

评审前必须读取：

- `AGENTS.md`
- `docs/ai-workflow.md`
- `PROJECT_STATUS.md`
- 当前任务单
- PR 描述
- 实际 diff
- CI 结果或本地验证结果

## 一级检查：任务有效性

| 检查项 | 失败结论 |
|---|---|
| 没有任务单 | `NEEDS_HUMAN` |
| 任务单仍是 Draft | `NEEDS_HUMAN` |
| 没有验收标准 | `NEEDS_HUMAN` |
| 没有明确不做 | `NEEDS_HUMAN` |
| PR 描述与任务不一致 | `REQUEST_CHANGES` |
| 实际 diff 与 PR 描述不一致 | `REQUEST_CHANGES` |

## 二级检查：实现正确性

- 是否实现任务目标。
- 是否遗漏验收标准。
- 是否只改了允许范围。
- 是否存在隐式副作用。
- 是否有清晰错误处理。
- 是否能重复运行。
- 是否能失败恢复。
- 是否可观测：日志、Trace、审计、状态。

## 三级检查：视频复刻 Agent 特有风险

### 视频分析

- 输出是否结构化、版本化。
- 是否保留证据帧、时间戳、置信度。
- 是否区分事实、推断、用户主观判断。
- 是否允许人工修正 VideoSpec。

### 分镜与故事板

- 是否先生成 `StoryboardSpec`，再按 Provider 渲染。
- 豆包是否使用完整镜头/场景级素描故事板。
- 豆包是否避免可识别真人脸。
- 可灵真实角色参考是否记录授权、作用域、来源。
- 首帧、尾帧、跨分镜连续性是否有版本记录。

### Prompt 生成

- Prompt 是否由结构化输入编译，不是自由拼接。
- 是否区分豆包、可灵 Provider 策略。
- 是否有 PromptVersion、模型版本、参数记录。
- 是否保存生成原因和证据来源。

### 反馈与知识库

- 用户反馈是否属于个人、项目、团队或全局作用域。
- 未审核反馈是否禁止进入团队知识库。
- 知识晋升是否可追踪、可回滚。
- RAG 检索是否有来源和命中记录。

### 多用户

- 是否服务端校验 Workspace / Team / Project / User。
- 是否避免跨用户读取素材、反馈、提示词。
- 是否有并发任务隔离。
- 是否有任务队列、公平调度、限流。

## 四级检查：测试

至少检查：

- 正常路径测试。
- 失败路径测试。
- 权限隔离测试。
- 幂等或重复提交测试。
- Provider Adapter mock 测试。
- 结构化输出 Schema 测试。

如果测试缺失但任务风险较高，必须 `REQUEST_CHANGES`。

## 结论规则

### APPROVE

满足：

- 任务有效。
- 验收标准通过。
- 无 P0/P1。
- 测试或验证证据充分。
- 无明显范围外修改。

### REQUEST_CHANGES

出现：

- 核心功能错误。
- 验收标准失败。
- CI 失败。
- 重大测试缺失。
- 范围外修改。
- 权限、数据、Provider 调用风险。

### NEEDS_HUMAN

出现：

- 任务目标不清。
- 验收标准不可判断。
- 需要业务决策。
- 需要成本、授权、合规、上线策略确认。
- 文档之间相互冲突。

## 评审用语要求

- 用事实描述问题。
- 指出文件和位置。
- 说明为什么违反任务或风险规则。
- 给出可执行修改建议。
- 不写泛泛的“建议优化”。
